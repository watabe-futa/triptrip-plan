# Doc-HR-001: TripTrip組織設計・組織文化戦略

## エグゼクティブサマリー

本文書は、TripTripの組織設計と組織文化戦略を包括的に定義します。15人規模のスタートアップから150人以上のグローバル組織への成長を見据え、Spotify、Google、Netflix、Metaの組織設計ベストプラクティスを統合した戦略を策定します。スクワッド/トライブ/チャプター/ギルドモデルの適用、リモートファースト/ハイブリッド勤務体制、組織文化の定義とカルチャーコード設計、意思決定プロセスとガバナンス構造を詳細に規定します。組織の成長フェーズごとの課題を先読みし、スケーラブルかつアジャイルな組織基盤の構築を目指します。

「組織はそのコミュニケーション構造を反映したシステムを設計する」

```yaml
conways_law_application:
  current_architecture:
    type: "モジュラーモノリス（Feature-based Package）"
    team_implication: "単一チームでの開発に最適"

  target_architecture:
    type: "マイクロサービス（ドメイン駆動）"
    team_implication: "独立したスクワッドによる並行開発"

  inverse_conway_maneuver:
    principle: "目標アーキテクチャに合わせて組織を先に設計する"
```

#### 3.1.2 バリューの行動指針

各バリューを具体的な行動に落とし込みます：

```yaml
value_behaviors:
  joy_of_discovery:
    do:
      - "新しいアイデアを積極的に提案する"
      - "技術カンファレンスやミートアップに参加する"
      - "実験的なプロジェクトに挑戦する"
    dont:
      - "「前例がないから」と新しいことを避ける"
      - "失敗を恐れて挑戦しない"
      - "学習の機会を逃す"

  ownership:
    do:
      - "自分の担当領域を超えて問題に取り組む"
      - "障害・インシデントに主体的に対応する"
      - "チームの意思決定に積極的に参加する"
    dont:
      - "「それは私の仕事ではない」と言う"
      - "問題を報告するだけで解決しようとしない"
      - "他人任せにする"

  user_first:
    do:
      - "機能開発前にユーザーの課題を理解する"
      - "ユーザーメトリクスを定期的に確認する"
      - "ユーザーフィードバックを積極的に収集する"
    dont:
      - "技術的な「かっこよさ」を優先する"
      - "ユーザーの声なしに機能を決定する"
      - "使いにくいが技術的に優れた解決策を選ぶ"
```

#### 5.1.2 RAPIDフレームワーク

重要な意思決定には、より明確なRAPIDフレームワークを使用します。

```yaml
rapid_framework:
  description: "Recommend, Agree, Perform, Input, Decide"

  roles:
    recommend:
      description: "提案を作成し、インプットを収集"
      responsibility: "分析、代替案の提示"

    agree:
      description: "提案に同意が必要な人"
      responsibility: "合意なしには進められない"

    perform:
      description: "決定を実行する人"
      responsibility: "実装、デリバリー"

    input:
      description: "インプットを提供する人"
      responsibility: "専門知識、意見の提供"

    decide:
      description: "最終決定を下す人"
      responsibility: "1人のみ、決定の説明責任"

  example_architecture_decision:
    context: "マイクロサービス移行の決定"
    rapid:
      recommend: "アーキテクトチーム"
      agree: "セキュリティリード、インフラリード"
      perform: "各スクワッドエンジニア"
      input: "全エンジニア、運用チーム"
      decide: "CTO"
```

#### 5.2.2 会議ルール

```yaml
meeting_rules:
  before_meeting:
    - アジェンダを24時間前に共有
    - 参加者リストを確認
    - 必要な資料を事前共有
    - 会議の必要性を再確認（非同期で可能か？）

  during_meeting:
    - 時間通りに開始、終了
    - 議事録担当を決定
    - タイムキーパーを設定
    - 発言機会を均等に

  after_meeting:
    - 24時間以内に議事録共有
    - アクションアイテムを明確化
    - フォローアップ責任者を指定

  no_meeting_zones:
    - "水曜午後": ノーミーティングデー
    - "金曜15時以降": ラップアップタイム
```

### 5.3 情報の透明性とドキュメンテーション

#### 5.3.1 透明性の原則

```yaml
transparency_principles:
  default_open:
    rule: "情報は原則公開、例外的に制限"
    open_by_default:
      - 技術的な意思決定
      - プロジェクト進捗
      - 組織の方向性
      - ミーティング議事録

    restricted:
      - 個人の人事情報
      - 給与・報酬詳細
      - 法的に機密な情報
      - セキュリティ脆弱性（修正前）

  decision_transparency:
    requirement: "重要な決定は背景を含めてドキュメント化"
    template:
      context: "なぜこの決定が必要か"
      options: "検討した選択肢"
      decision: "選んだ選択肢"
      rationale: "選んだ理由"
      consequences: "予想される影響"
      owner: "決定者"
      date: "決定日"
```
